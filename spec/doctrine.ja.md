# LLM開発のための情報統治設計書

本書は、受領するClaude（主にClaude Code）へのビルド指示書である。ここで定義する情報統治を、再利用できるClaude Codeプラグイン（Skills・Hooks・スクリプト・テンプレート）として実装させる。

本書自身は `spec/` に置く上位設計書であり、`docs/` の登録簿による統治の対象外である。`type: DOCTRINE` はこの唯一の設計書を指し、§3.2 の型登録簿には載せない。

本書の規約。実装する体系にも同じ規約を課す。

- 用語: 統制語彙（§1）の承認語だけを使う。一語一義。禁止同義語は使わない。
- 最小性: 要求（§2）に紐づかない記述を書かない。テストできない記述は弾く。どこからも依存されない文書は取り除く。逆に、必要なのに欠けた文書・記述も残さない。過剰と不足の両方を見る。
- 全体図と索引: モデル（§3の登録簿）から描画する投影とし、手で保守しない。
- 履歴: 現行と履歴を分け、履歴はIDで引く。本書も本文に改訂履歴を書かない。
- 日本語: 訳語臭（カルク）を入れない。英語の語・比喩・構文・一語訳をなぞらない。一文一義で、冗長を削る。

`[R番号]` は、その記述が満たす要求（§2）を指す。

## 1. 統制語彙　モデル

用語辞書の正本である。各語は一つの意味だけを持つ。右列の同義語は使わない。実装するGlossaryは、この辞書の運用版である。`[R6]`

| 承認語 | 唯一の意味 | 禁止する同義語 |
|---|---|---|
| 文書 | 管理対象の最小単位（S1000Dのデータモジュールに当たる） | ドキュメント、資料、ページ |
| ファイル | ディスク上の実体。「文書」の意味では使わない | （文書の意味でのファイルを禁止） |
| ドメイン | 交換できる区画（LRU／ARINC 653パーティションに当たる） | 領域、サブシステム、コンテキスト |
| ICD | ドメインが公開するインターフェース定義文書 | IF、インターフェース（単独語）、接続仕様 |
| 境界 | ドメインとドメインの接点 | つなぎ目、接合部 |
| 正本 | ある事実の唯一の権威ある出所 | マスター、原本、オリジナル |
| 投影 | モデルから描画した派生表示。手で保守しない | ビュー、出力、レンダリング結果 |
| 現行 | いま有効な版（status=current/accepted） | 最新、現在、アクティブ、有効 |
| 履歴 | 過去の状態の記録 | 経緯、過去、来歴 |
| 廃止 | 方針を撤回した状態（status=deprecated） | 廃案、ボツ |
| 置換 | 後継で入れ替えた状態（status=superseded） | 差し替え（操作名としては可。状態名は置換） |
| アーカイブ | 証跡として倉庫へ退避した状態（status=archived） | 退役、保管 |
| 要求 | 文書または機能が存在してよい理由 | 要件、ニーズ |
| 依存 | 文書が他文書を前提とする関係（depends_on） | （リンクの意味の「参照」とは区別する） |
| 参照 | 文書から別文書への指し示し（リンク）。依存とは限らない | （依存と混同しない） |
| 孤児 | どの現行文書からも依存されない文書 | 浮き、迷子 |
| 逆孤児 | あるべき文書の不在。対応する仕様を持たない要求、対応するテストを持たない受入基準など。孤児の対称 | 抜け、漏れ |
| ガード | 違反を機械的に拒否するHook | （ガードで統一） |
| リンタ | 文書を機械的に点検するスクリプト | （用語チェッカーはリンタの一機能） |
| 用語チェッカー | リンタのうち、未承認語と未定義語を弾く機能 | （上記で統一） |

承認語を増やす、または意味を変えるときは、ADRで決め、この表を更新する。各文書はこの表を二重定義しない。`[R6][R8]`

**禁止表現（カルク辞書）**。承認語に加えて、英語の比喩・構文・一語訳をなぞった表現も使わない。`[R10]`

| 使わない（カルク） | 直す | なぞった英語 |
|---|---|---|
| 同じページにいる | 認識を揃える | on the same page |
| テーブルの上にある | 検討中 | on the table |
| 針を動かす | 効果を出す | move the needle |
| 一日の終わりには | 結局 | at the end of the day |
| 〜を可能にする（過剰） | 〜できる | enable |
| 〜の文脈で（過剰） | 〜では／〜について | in the context of |
| 深く潜る／掘り下げる | 詳しく見る | deep dive |
| 予算（資源量の意味で） | 上限・割り当て | budget |
| ロールアップ | 統合・一本化 | roll up |

一語訳の罠（辞書の一番目を疑う）: status→位置づけ・区分、native→標準で・組み込みで、robust→壊れにくい、leverage→活かす。

判定は逆翻訳テル。怪しい箇所を英語に戻し、英語の慣用句・固定表現にそのまま戻れば、それはカルク。直すときは英語を捨て、日本語で選び直す。定着した借用語（データ・リスク等）と普通の否定文はカルクではない（擬陽性を避ける）。強制は二段。リンタが上の禁止表現を機械的に照合する。一覧にない訳語臭は doc-review が判定する。新しいカルクは、別ファイルを作らず、この表に一行足す。

**文章の規則**（doc-review が点検する）。`[R10]`

- 一文一義。係り受けが二通りに取れる文は短く割る。
- サ変名詞＋「を行う」を動詞にする（「初期化を行う」→「初期化する」）。「することができる」→「できる」。
- 専門語・略語は初出で定義する。
- 抽象語は具体語・数値・固有名に置き換える。
- 同じ概念は同じ語で書く（この用語辞書）。

## 2. 要求

体系が満たす要求を列挙する。各要求はテストできる形で書く（§6が対応するテストを持つ）。テストできない願望は要求として認めない。

- **R1 見つけやすさ**: 目的の文書に到達できる。
- **R2 現行性**: どれが現行かを機械的に判別できる。確定事実にも期限を持たせ、古びを検出できる。
- **R3 追跡性**: 要求→仕様→実装→テスト→決定を依存リンクでたどれる。逆方向（要求に対応する仕様、受入基準に対応するテスト）の不在も検出できる。
- **R4 変更耐性**: 変更時に更新すべき文書集合を依存から列挙できる。
- **R5 LLM適合**: 常時投入を最小に保ち、廃止文書の本文をLLMに渡さない。常時投入には上限を設け、超えたら縮める。
- **R6 用語統一**: 同じ概念を同じ語で書く。未承認語・禁止同義語を弾く。
- **R7 境界明瞭**: ドメイン間の依存はそのドメインのICD宛だけ許す。内部宛の依存は拒否する。
- **R8 最小性**: 要求に紐づかない文書を作らない。テストできない記述を弾く。孤児を取り除く。重複した事実を一本化する。かつ、必要なのに不在の文書・記述を残さない（逆孤児を検出する。逆方向の追跡性）。
- **R9 保証限界の明示**: 何を予防・検出・委ねるかを各成果物に書く。
- **R10 明快な日本語**: 訳語臭（カルク）を排し、一文一義で書く。英語の語・比喩・構文・一語訳をなぞらない。

R7が中心の追加である。R8は「以上でも以下でもない」の両側を見る。

## 3. 全体モデル

本章は体系の単一の正本である。実装はすべて本章の登録簿に従う。登録簿の変更は本章でだけ行う。コードに規則を二重定義しない。

### 3.1 ドメイン登録簿

体系は、システム階層と複数のドメインに分かれる。ドメインは交換できる区画であり、自分の用語と自分のICDを持つ。`[R7]`

- **システム階層（`_system`）**: 横断の正本と投影を置く。用語辞書（Glossaryの正本）、Decided Facts、Non-goals、Watchlist。投影として Overview、Context Map、ICD一覧。
- **各ドメイン（`<domain>`）**: 一つの区画。例 `billing`、`identity`。内部に仕様・決定・実装・テスト・調査・アーカイブを持つ。公開する接点はICD一文書に集約する。

ドメインの粒度は、用語が一意に閉じる範囲で切る。同じ語が二つの意味を持つなら、ドメインを分ける合図である。`[R6]`

### 3.2 文書型登録簿

各文書は「型」を持つ。型コードはIDの接頭辞であり、置き場所と既定の `llm_context` を決める。リンタは型と置き場所と status の整合をこの表で点検する。`[R1][R2]`

| 型コード | 文書型 | 置き場所 | 既定status | llm_context | 入れる | 入れない |
|---|---|---|---|---|---|---|
| ICD | Interface Control（公開接点） | `<domain>/` | current | task | 公開する用語、正本である事実、他ドメインが依存してよいデータ契約 | 内部実装、内部の検討 |
| OVERVIEW | Overview（投影） | `_system/` | current | always | 現行文書一覧、各一行説明 | 仕様本文 |
| GLOSSARY | 用語辞書（正本） | `_system/` | current | always | 承認語、定義、禁止同義語 | 曖昧な言い換え |
| CTXMAP | Context Map（投影） | `_system/` | current | task | ドメインとICDの全体図、結合の要点 | 仕様本文 |
| DECIDED | Decided Facts | `_system/` | current | always | 確定方針、決定日、根拠ADRのID、`review_by` | 迷い、調査、提案中 |
| NONGOAL | Non-goal | `_system/` | current | always | やらないこと、理由 | 曖昧な願望 |
| WATCH | Regression Watchlist | `_system/` か `<domain>/test/` | current | always（要点） | 戻してはならない事項、撤回日、根拠、`review_by` | 安定機能 |
| REQ | Requirement | `<domain>/` | current | task | 要求文、優先度、受入基準参照、出所 | 実現方法 |
| SPEC | Specification | `<domain>/spec/` | current | task | 入出力・制約・エラー時挙動・受入基準（この4節を必須とする） | 廃止、検討、実装コードの写し |
| DATA | Data Model | `<domain>/spec/` | current | task | エンティティ、保存方針、保持期間 | 処理ロジック |
| API | API Reference | `<domain>/spec/` | current | task | エンドポイント、入出力、エラー（中立・現在形） | 「なぜ」の解説 |
| ADR | Decision Record | `<domain>/decisions/` | accepted | task | 背景、却下した選択肢、決定、帰結（1文書1決定） | 複数決定、現行仕様の全文 |
| CHANGE | Change Request | `<domain>/decisions/` | proposed | task | 変更内容、理由、要求元、影響の初期見積 | 承認前の決定 |
| IMPACT | Impact Analysis | `<domain>/decisions/` | current | task | 影響する文書・実装・テスト、工数見積 | 感想 |
| IMPL | Implementation Note | `<domain>/implementation/` | current | task | 実装制約、注意点、対象部品 | 仕様の正本 |
| TEST | Test Plan | `<domain>/test/` | current | task | 受入基準への対応、退行観点、合否基準 | 無関係な要求 |
| RESEARCH | Research Note | `<domain>/research/` | draft | never | 出所、取得日、事実、比較 | 決定 |
| ARCHIVE | Archive Note | `<domain>/archive/` | archived | never | アーカイブ理由、アーカイブ日、後継ID | 現行情報 |

新しい型を増やすのは、既存型で表せない情報が出てからにする。空の型を先に作らない。`[R8]`

### 3.3 ステータス統制語彙

| status | 意味 | 主な型 |
|---|---|---|
| proposed | 提案中 | CHANGE |
| accepted | 受理（ADR専用）。現行に相当 | ADR |
| current | 現行 | 大半の型 |
| deprecated | 廃止。現行ではない | 廃止した各型 |
| superseded | 後継で置換した | ADR、SPEC旧版 |
| archived | アーカイブ。証跡 | ARCHIVE |
| open | 未決 | 未決事項を扱う文書 |

ADRの許可値は {proposed, accepted, superseded, deprecated}。それ以外の型は accepted を除く6値。リンタは型ごとの許可リストで点検する。`[R2]`

### 3.4 メタデータ・スキーマ

全文書の先頭にYAMLフロントマターを置く。必須は絞る。`[R3]`

```yaml
---
id: <型コード>-<連番>          # 例 SPEC-014。ファイル名と一致させる
title: <一文の主題>
type: <§3.2の型コード>
domain: <§3.1のドメイン名>       # どの区画の文書か。_system はシステム階層
status: <§3.3の語彙>
owner: <個人名>                # 陳腐化の責任の所在
created: <YYYY-MM-DD>
updated: <YYYY-MM-DD>          # 現行性の点検に使う
sources: [<出所URL/会話ID>]
# --- Level 3+ で推奨 ---
review_by: <YYYY-MM-DD>        # 次回点検期限。DECIDED・WATCHでは必須
depends_on: [<id>]            # 前提とする文書。R7の対象
impacts: [<id>]               # 変更時に波及する先
superseded_by: <id>           # 後継（退行防止の要）
canonical_for: [<トピック識別子>]  # この文書がそのトピックの正本である宣言
llm_context: <always|task|never>
---
```

リンタが必須とする項目（Level 2以降）: `id, title, type, domain, status, owner, updated, sources`。DECIDED・WATCHでは `review_by` も必須とする（古びの検出に使う）。`canonical_for` はLevel 4+で任意とし、同じ事実が二重に書かれた型から付ける。Level の段差は§4.4の段階導入に対応する。`[R6][R8]`

### 3.5 ICD

各ドメインは、外から依存してよいものを `<domain>/ICD.md`（型=ICD）一文書に集約して公開する。ICDが宣言するのは三つ。`[R7]`

1. **公開する用語**: そのドメインが所有し、他ドメインが使ってよい語（用語辞書の該当分への参照）。
2. **正本である事実**: そのドメインが唯一の正本である事実（`canonical_for` のトピック）。
3. **データ契約**: 他ドメインが依存してよい入出力・スキーマ。

ICDの内部実装は書かない。ICDの中身が変わるときは、依存する全ドメインの合意が要る。

### 3.6 依存規則

- **ドメイン内の依存**: 自由。同じドメインの現行文書どうしは `depends_on` で結んでよい。
- **ドメイン間の依存**: 相手ドメインのICD宛だけ許す。相手の内部文書（SPEC・IMPL・DATA など）を直接 `depends_on` してはならない。
- **強制**: 違反する `depends_on` はガードが拒否する（§4.2）。これにより、ICDを変えない限り、内部の変更は境界を越えない。内部が古くなっても、ドメインごと差し替えられる。

### 3.7 配置

```
<repo>/
├── AGENTS.md            # 投影。最小の案内。手で保守しない
├── CLAUDE.md            # 投影。同上
└── docs/
    ├── _system/         # システム階層（横断）
    │   ├── glossary.md          # 用語辞書（正本）
    │   ├── decided-facts.md     # 正本
    │   ├── non-goals.md         # 正本
    │   ├── watchlist.md         # 正本（要点）
    │   ├── overview.md          # 投影（描画）
    │   ├── context-map.md       # 投影（描画）
    │   └── icd-index.md         # 投影（描画。全ICD一覧）
    └── <domain>/        # ドメイン（区画）。例 billing
        ├── ICD.md              # 公開接点（正本）
        ├── spec/               # SPEC/DATA/API
        ├── decisions/          # ADR/CHANGE/IMPACT
        ├── implementation/     # IMPL
        ├── test/               # TEST/WATCH
        ├── research/           # RESEARCH
        └── archive/            # アーカイブ（現行と分離）
```

`docs-system-init` の既定は全ツリーを作らない。`_system` の最小（glossary・decided-facts・non-goals・overview の投影）とルートの案内だけを置く。ドメインのフォルダと各層は、その型の文書が最初に作られるときに `doc-author` が生成する。空の足場を先に作らない。`[R8]`

命名: `<型コード>-<連番>-<短い主題>.md`（例 `SPEC-014-refund-policy.md`）。`id` はファイル名と一致させる。日付はISO 8601。日本語ファイル名・空白・版番号の埋め込みは禁止。

### 3.8 ライフサイクル

**削除は破壊ではなく一段降格とする**。各段は、後から引ける記録（後継ID付き）を残す。原則は可逆である。`[R5][R8]`

1. 現行（current）。
2. 廃止（deprecated）。方針を撤回。事実だけをDECIDEDの対の記録に残し、本文はLLMに渡さない。
3. アーカイブ（archived）。証跡のある本文を `<domain>/archive/` へ移し `superseded_by` を付ける。検索から除外。編集はガードが拒否。
4. git履歴のみ。文書は消すが、コミットに撤去理由とIDを残す。
5. 完全消去。孤児かつ証跡なしかつ再現できる一時文書だけに限る。

**降格の不変条件**: 段を下げる前に、その文書への現行の依存がゼロであることを確認する（逆参照ゼロ）。依存が残るなら、同じ変更で参照元を後継へ張り替えてから下げる。ガードが強制する（§4.2）。`[R4]`

**確定事実の古び対策**: DECIDED・WATCHの確定事実にも `review_by` を付け、期限で再点検する。置換済みの決定は、要点だけを残して一本の事実に統合する。常時集合（always）は確定事実が積もるため、統合と期限切れの整理で縮める。これは§3.9の注入量の上限と対になる。`[R2][R5]`

**変更フロー**: 根拠（ADR）→現行仕様（SPEC）→実装→テスト→LLMコンテキスト→廃止整理。順序を守る。「何を変えたか」（CHANGE／git）と「なぜ決めたか」（ADR）を分ける。`[R3][R4]`

**現行と履歴の分離**: 現行文書は最小に保つ。変更の理由は本文に書かず、ADRのIDで引く。`[R8]`

### 3.9 投影

次の文書は投影である。正本（本章の登録簿と各文書のフロントマター）から描画する。手で書き溜めない。`[R1][R8]`

- **Overview**: 現行文書の一覧と一行説明。フロントマターから決定論で描画する。
- **ICD一覧**: 全ドメインのICDの索引。ICD文書から描画する。
- **Context Map**: ドメインとICDの全体図。依存グラフから描画する。
- **SessionStart契約**: LLMへ常時渡す最小の事実。DECIDED・NONGOAL・WATCH・廃止事実から描画する（§4.2）。

索引型の投影（Overview・ICD一覧）は決定論で描画できる。意味を含む投影（用語定義・Context Mapの結合説明）は `docs-curate` が補助し、リンタはずれ（投影と現行集合の差）を検出する。

集約とは、これらの投影を正本から**描画し直す**ことであり、別途要約を書き溜めることではない。SessionStart契約も、正本の文書群を、重要な事実だけに絞った投影である。絞り込みの質は、まず取りこぼしを減らし、次に重複を削る順序で上げる。`[R5]`

**最小コンテキストの作り方**。常時集合と各タスクのコンテキストは、次の規則で小さく保つ。長い入力はそれ自体がLLMの成功率を下げるため、量と紛れの両方を抑える（§7・付録Cに根拠）。`[R5]`

- **注入量の上限**: 常時集合の注入量（トークン数）に上限を設ける。超えたら `docs-curate` を起動し、統合と期限切れの整理で縮める。上限は肥大を機械的に検出する歯止めである。
- **要点の復唱**: 注入の冒頭で要点を先に書き出させる（復唱）。長文による劣化を抑える。
- **最少集合のパック**: タスク別のコンテキストは、要求の被覆を満たす最少の文書集合に絞る。関係しない文書を足さない。
- **出所の表示**: パック内の各事実に、由来する文書（出所）を表示する。語彙が近い文書の間で取り違えを防ぐ。
- **位置の配慮**: 重要な文書は、注入の冒頭と末尾の両方に置く。端ほど注意が高い性質に合わせる。

## 4. 実装

役割分担はこうする。決定論で守れるものはHook（ガード／リンタ）が守る。判断が要るものはSkillが支援する。検証はスクリプトが担う。`[R9]`

### 4.1 Skills

Skillの本文は500行未満。詳細は `references/` に分ける。決定論処理は `scripts/` に委ねる。descriptionは三人称で書き、起動を促す表現と実際に使う語句を含める。

| Skill | 役割 | 主な要求 |
|---|---|---|
| `docs-system-init` | `_system` の最小配置、案内、ガード/リンタ設定。既存を壊さない | R1,R8 |
| `doc-author` | 型付き文書（ICDを含む）の作成・更新。型・置き場所・フロントマターを正す | R1,R3,R6,R7 |
| `doc-review` | 文章規範と位置づけのレビュー。用語チェッカーを併走（未承認語・禁止同義語・禁止表現・未定義語）。カルクは逆翻訳テルで判定する。doc-authorが著述のたびに起動する（定例だけに頼らない）。指摘は定義の在処へ書き戻す（一覧外カルク→§1表、新承認語→ADRと用語辞書）。canonical_for未付与・辞書外の訳語臭・意味的重複は定例（周期を定める）で点検する | R6,R10 |
| `change-impact` | 14ステップの変更フロー（全14ステップは change-impact が定める）。依存をたどり影響を列挙。更新順序を守る | R3,R4 |
| `regression-guard` | 廃止の復活・撤回方針の再採用を防ぐ。DECIDED・WATCHと突き合わせる | R5 |
| `llm-context-pack` | タスク別の最小コンテキストを集約。never群を除外。被覆を満たす最少集合に絞り、出所を表示する | R5 |
| `docs-curate` | 点検→統合／降格／削除を一片ずつ。逆参照を確認。投影を描画し直す。常時集合が上限を超えたら縮める | R8 |

Skillは7つに限る。`doc-author` がICDも書く（ICD専用Skillは作らない）。集約と削除は一活動なので `docs-curate` 一つにまとめる。`[R8]`

### 4.2 Hooks

per-turnのHookは単一文書だけを点検する。全件走査は監査（セッション境界のHookとCI）へ隔離する。

**SessionStart → 契約の最小注入**。`inject-contract.py` を実行し、DECIDED（現行）・NONGOAL・廃止事実・GLOSSARY見出し・WATCHの要点だけを描画して渡す。本文全量は渡さない。注入は§3.9の規則に従う。すなわち、注入量の上限を守り、冒頭で要点を復唱させ、重要な文書を冒頭と末尾に置く。上限を超えるなら、その旨と `docs-curate` の起動を促す。あわせて、前回の監査結果の要約を注入する。`[R5]`

**PreToolUse → 三つのガード**（matcher: `Edit|Write|MultiEdit`、Bashは別マッチャ）。`[R7][R8]`

1. **不変性ガード**: `<domain>/archive/**` の編集と、既存ADRの改変を拒否する。理由を返す。
2. **ICD依存ガード**: 書き込み内容のフロントマターを読み、`depends_on` の各IDのドメインを引く。自ドメイン外で、かつ相手のICD以外を指す依存があれば `permissionDecision: "deny"`。理由に違反する依存と「ICD宛にせよ」を載せる。Writeは内容が全量来るので事前に拒否できる。Edit/MultiEditは内容の全量が来ないため事前に拒否できず、適用後に再読して違反なら `decision: "block"` で止めて指摘する。block は実行を取り消せず、修正までディスクが一時的に不整合になるため、可能な編集はWrite経路に寄せて実行前拒否に倒す。
3. **削除安全ガード**: 現行文書を廃止・置換・アーカイブへ降ろす編集、本文を空にする編集、Bashの `rm`／`git rm`／`mv`（matcher: `Bash`, if: `Bash(rm:*)` 等）で、その文書への現行依存が残るなら拒否する。逆参照は `dep-graph.py` を逆方向で呼ぶ。Bashマッチャでは文脈注入（additionalContext 等）が現状モデルに届かないため、このガードは `permissionDecision: "deny"` だけに頼る（§7）。

ICD依存ガードの判定（擬似仕様）:
```
proposed = parse_frontmatter(tool_input.content)         # Write の場合
self_domain = proposed["domain"]
for dep in proposed.get("depends_on", []):
    dep_domain = domain_of(dep)                          # 登録簿から引く
    if dep_domain != self_domain and type_of(dep) != "ICD":
        deny(f"{dep} は {dep_domain} の内部です。{dep_domain} の ICD 宛にしてください。")
```

**PostToolUse → リンタ**。`docs-linter.py` を実行する。点検は、必須キーの存在、status の型別許可、id とファイル名の一致、type と置き場所の整合、`llm_context` の値、型規律（RESEARCHに「決定」見出しがあれば警告）、**SPEC必須4節の存在（入出力・制約・エラー時挙動・受入基準。空節は指摘）**、**用語チェッカー（未承認語・禁止同義語・禁止表現（カルク辞書）・未定義語）**、**ICD依存規則の照合（事後の検出として）**、**追跡性（SPEC/IMPL/TEST が要求または依存を持つか）**。違反は `decision` を出さず `additionalContext` で指摘し、Claudeに自己修正させる。参照整合（dead link）と逆方向の追跡性は全件走査が要るので監査へ。`[R3][R6][R7][R8][R10]` あわせて `review-nudge.py` を実行し、型付き文書の編集に doc-review（文章規範・一覧外カルク・位置づけ）を促す（助言のみ。`decision` は出さない。doc-author を介さない手編集を補う。§4.1）。`[R10]`

**監査（セッション境界のHookとCI、毎ターン実行しない）**。`docs-audit.py` が全件で、dead link、`review_by` 超過（DECIDED・WATCHを含む）、draft放置、孤児（逆参照ゼロ∧陳腐化∧再現可能）、**逆孤児（要求に対応する仕様の不在、受入基準に対応するテストの不在。逆方向の追跡性）**、`canonical_for` 衝突、語彙的に酷似する文書対（助言）、ICD依存違反、投影ドリフトを一覧化する。起動は人手のコマンドに頼らない。SessionEnd のHookとCIから走らせ、結果は次の SessionStart で要約を注入する。毎ターンは走らせない（体感速度を守る）。周期はこの二経路で定める。`[R1][R3][R4][R8]`

### 4.3 スクリプト

外部pip依存を作らない。フロントマターはフラットなので最小パーサで読む。

| スクリプト | 呼び出し元 | 役割 |
|---|---|---|
| `_frontmatter.py` | 共有 | フロントマター解析 |
| `_registry.py` | 共有 | 型・status・llm_context・必須キー・置き場所の登録簿（規則の単一の出所） |
| `_depgraph.py` | 共有 | 依存グラフの中核。`dep-graph.py` と監査が読み込む |
| `_termcheck.py` | 共有 | 用語チェックの中核（辞書の解析・照合）。`term-check.py` とリンタが読み込む |
| `docs-linter.py` | PostToolUse | 単一文書の点検（SPEC必須節・用語チェッカー・ICD照合・追跡性を含む） |
| `review-nudge.py` | PostToolUse | 型付き文書の編集時に doc-review を促す助言（`decision` は出さない） |
| `policy-guard.py` | PreToolUse | 不変性・ICD依存・削除安全の三ガード |
| `inject-contract.py` | SessionStart | 最小契約を描画して注入（注入量の上限・要点の復唱・末尾配置・前回監査の要約） |
| `docs-audit.py` | SessionEnd/CI | 全件の孤児・逆孤児・dead link・`canonical_for`衝突・ICD違反・投影ドリフト・`review_by`超過 |
| `dep-graph.py` | change-impact, ガード | 依存の有向グラフ。波及先と逆参照を列挙。ドメイン跨ぎを分類。逆方向で逆孤児を出す |
| `render-projection.py` | docs-curate | Overview・ICD一覧・Context Map をフロントマターから描画 |
| `term-extract.py` | docs-curate | 各ドメイン（フォルダ）を1クラスとしたclass-based tf-idf（c-TF-IDF）でドメイン特徴語の候補を出す。辞書の素案づくり。採否は人間 |
| `term-check.py` | docs-linter | 禁止同義語・禁止表現（カルク辞書）・未定義語の照合 |
| `collect-context.py` | llm-context-pack | `llm_context` でフィルタした最小コンテキスト。被覆を満たす最少集合に絞り、出所を表示する |
| `scaffold.py` | docs-system-init | `_system` 最小配置（非破壊）。段階導入の縮小構成を選べる |

用語チェッカーは、辞書（§1）を発見しない。承認済みの辞書を強制する。禁止同義語マップ（承認語→禁止語）を持ち、禁止語の出現と、辞書にない専門語の初出を弾く。型コード（§3.2）と要求タグ（§2）は別の正本で定義済みのため、未定義語に数えない（登録簿を引く。辞書に二重定義しない）。カルクは、§1の禁止表現を機械的に照合し、一覧にない訳語臭は doc-review が逆翻訳テルで判定する。`term-extract.py`（c-TF-IDF）はその辞書の候補を出すだけで、確定は人間が承認する。「同じ概念を別の語で書いた」と「英語をなぞった表現」の完全な検出は埋め込みかLLMが要るため、構造（辞書＋禁止同義語＋禁止表現）で先回りする。`[R6][R10]`

### 4.4 段階導入

体系の重さを規模に合わせる。全部を最初から置かない。痛みが出た所だけ足す。これは体系自身の最小性である。`[R8]`

- **小規模（Level 2・縮小構成）**: 必須キー（§3.4）だけ。型は ICD・REQ・SPEC・ADR・DECIDED・OVERVIEW(投影) に絞る（DECIDED は `review_by` を持つ）。スクリプトは `_frontmatter.py`・`docs-linter.py`・`policy-guard.py`・`inject-contract.py` だけ。`scaffold.py` は `_system` の最小配置と `.docs-level` の目印を置く（Hook は事前生成しない）。縮小構成の Hook（`hooks.level2.json`）は同梱の別ファイルで、ホストが配線する。
- **中規模（Level 3）**: `depends_on`・`impacts` を加える。`dep-graph.py`・`change-impact`・`docs-audit.py` を足す（`review_by` 超過の点検はここから）。
- **大規模（Level 4）**: `canonical_for` と全件監査・投影一式・ドメイン連携を加える。

現に選んでいる Level は、文書のフロントマターではなく `docs/_system/.docs-level`（`level: N` の一行）に記録し、他のスクリプトがそこから読む。上位へ上げるのは、その情報が要るとわかってからにする。

## 6. 受け入れ基準

各要求に、確認するテストを対応させる。テストできない要求は置かない。

| 要求 | 合格を確認するテスト |
|---|---|
| R1 見つけやすさ | Overview投影が現行文書を網羅する。監査が孤児を検出する |
| R2 現行性 | リンタが status の型別許可を強制する。監査が `review_by` 超過（DECIDED・WATCHを含む）を一覧化する |
| R3 追跡性 | `dep-graph.py` が要求→仕様→実装→テスト→決定をたどる。リンタが要求なしSPEC/IMPL/TESTを指摘する。監査が逆方向（要求に対応する仕様、受入基準に対応するテストの不在）を検出する |
| R4 変更耐性 | `change-impact` が影響集合を列挙する。監査が dead link を検出する |
| R5 LLM適合 | `inject-contract.py` が事実・要点のみを注入する。never群が渡らない。注入量の上限を守り、超過時に `docs-curate` を促す。冒頭で要点を復唱する |
| R6 用語統一 | `term-check.py` が禁止同義語と未定義語を弾く。辞書（§1）を二重定義しない |
| R7 境界明瞭 | ICD依存ガードが、ドメイン外の非ICD宛 `depends_on` を拒否する（自己テストで実証） |
| R8 最小性 | 監査が孤児・逆孤児・テスト不能記述・`canonical_for`衝突を一覧化する。リンタが SPEC 必須4節の空を指摘する。`docs-curate` が一片ずつ取り除く |
| R9 保証限界 | 各成果物に予防・検出・委ねるの別が書かれている（本書§7を継承） |
| R10 明快な日本語 | `term-check.py` が禁止表現（カルク）を照合する。doc-review が逆翻訳テルで一覧外のカルクを判定する |

メタ条件: プラグインが配布できる。`docs-system-init` が既存を壊さない。スクリプトが標準ライブラリだけで動く。per-turnのHookがエージェントを体感的に遅くしない。監査が人手起動に頼らず、セッション境界のHookとCIから周期的に走る。縮小構成が小規模で破綻なく動く。ガードの変更がセッション開始時に反映される前提で運用される。

## 7. 既知の限界

できないことを明記する。`[R9]`

- 二つの記述が**意味として**重複か矛盾かの最終判断はできない。語彙的検出（c-TF-IDF・Jaccard）は言い換えを取りこぼす。`canonical_for` 衝突は確実だが、付与は人間に依る。最終判断は人間とLLMに委ねる。canonical_for未付与・辞書外の訳語臭・意味的重複は doc-review の定例でしか閉じないため、これを運用契約として周期に定める。
- 不足の検出は、**構造的な欠落**（指されるべき相手の不在、要求→仕様→テストの対応の不在）に限る。一度も記録に現れなかった必要（書かれなかった要求）は検出できない。
- 用語チェッカーは文意の良し悪しを判定できない。承認語の強制までに限る。
- カルクの機械検出は、§1の禁止表現の一覧分だけを照合する。一覧にない訳語臭は doc-review（逆翻訳テル）で拾うが、完全な検出はできない。一覧を育てて低減する。
- ICD依存ガードと削除安全ガードは、よくある経路（Edit・rm・git rm・mv）を拒否する。任意スクリプト経由の回避は防げない。監査の逆参照が、事後に検出して補う。
- Edit/MultiEditの境界違反は事後検出であり、修正までディスクが一時的に不整合になる。Write経路に寄せて予防の比率を上げる。
- Bashマッチャの Hook では文脈注入（additionalContext・素のstdout 等）が現状モデルに届かない。削除安全ガードは拒否（deny）だけに頼る。
- 常時集合の縮小は注入量の上限で機械的に促すが、何を残すかの最終判断は人間に依る。
- Hookの設定はセッション開始時にスナップショットされ、即時反映されない。ガード変更は新しいセッションで効く。
- 仕様が実装どおり動くかは、テストの責務であり、構造だけでは閉じない。
- 常時注入を最小に保つことの性能上の益は、長文ほど劣化するという知見と整合するが（付録C）、最適な上限は運用と受入テストで確かめる。肥大はLLMの成功率を下げる。
- 「100%の予防」は構造上不可能である。本体系の効果は、適切に運用された場合に特定の失敗類型を検出・早期発見できることに限る。

```markdown

id: ICD-<連番>
title: <ドメイン名> のインターフェース
type: ICD
domain: <ドメイン名>
status: current
owner: <個人名>
created: <YYYY-MM-DD>
updated: <YYYY-MM-DD>
sources: []
canonical_for: [<このドメインが正本である事実の識別子>]
---

# <ドメイン名> ICD

## 公開する用語
（このドメインが所有し、他が使ってよい語。用語辞書の該当分を参照）

## 正本である事実
（このドメインが唯一の正本である事実。canonical_for と一致させる）

## データ契約
（他ドメインが依存してよい入出力・スキーマ・更新頻度）
```

## 付録B. 他テンプレート

DECIDED・NONGOAL・ADR・SPEC・WATCH・RESEARCH・ARCHIVE・OVERVIEW（投影）・GLOSSARY（辞書）の各テンプレートは `templates/` に置く。フロントマターは§3.4に従う。DECIDED・WATCHのテンプレートは `review_by` を必須欄として持つ。SPECのテンプレートは必須4節（入出力・制約・エラー時挙動・受入基準）を見出しとして持つ。投影テンプレートは「描画される。手で編集しない」を冒頭に明記する。

## 付録C. 出典
