# リンタは登録済み非文書と統治木外に schema 強制をしない（用語助言のみ）

## 背景

SPEC-007 §エラー時挙動は「フロントマターが無い → 必ず `MISSING_FRONTMATTER`」を無条件に規定していた。これは「全 .md は文書かノイズ」という、非文書が正規化される前の前提である。ADR-021 が intake（`.md-intake` に 非文書／投影／保留 を記録する）を定めた後も、リンタはこの記録を参照せず、監査（docs-audit）が非文書と認めたファイル（README・各 skill の SKILL.md・参照散文・spec/doctrine.ja.md など実測 37 件）に編集のたび schema/frontmatter の ERROR を出し続けた。統治木の根に到達できない体系外のファイル（ハーネスのメモリなど）にも同様に発火した。監査（全体を見る）とリンタ（一件を見る）が同じファイルへ矛盾した判定を出す状態であり、正規状態が恒常的に「要修正」と報告されて警報の信頼が摩耗し、エージェントを `.txt` 改名などの設計違反の回避へ誘導した。[R2][R7]

## 却下した選択肢

- リンタ側に独自の intake 読み取りを書く。監査と別実装になり、いつか書式の解釈がずれて同じ食い違いが再発する（本件の欠陥クラスそのもの）。
- `MISSING_FRONTMATTER` を一律に廃止する。未登録の統治木内文書でのフロントマター欠落という本物の違反まで見逃す。
- 監査だけを直し、リンタの助言ノイズは運用で無視する。正規状態への ERROR が常態化し、本物の違反が埋もれる状態が残る。ADR-017 が擬陽性の害として却下した論法と同じである。

## 決定

リンタ（`docs-linter.py`）は判定の前に統治木を walkup で探し、次の順で扱う。①統治木が無ければ（体系外）何も出さない。②型付き（登録簿の型を持つ）文書は従来どおり全点検し、統治木の外なら `STRAY_DOCUMENT` を出す。③型なしで intake に「非文書」または「投影」と登録されたファイルは、schema/frontmatter の点検を飛ばし、用語助言だけを WARN で残す（ERROR は出さない）。④それ以外（型なし・未登録）は従来どおり（フロントマター無しは `MISSING_FRONTMATTER`）。intake の読み取りは監査と共有する単一のコア（`_intake.py`）へ一本化し、両者が同じ記録・同じ書式解釈を使うことで食い違いを構造的に防ぐ。この非文書は本当に非文書か、という判断は従来どおり docs-curate（人の判断層）に委ねる。

## 帰結

監査が非文書と認めたファイルにリンタが ERROR を出さなくなり、両者の判定が一致する（整合点検 `scripts/consistency-check.py` が緑になる）。LLM に全文注入されるプロンプト資産の .md や、ハーネスが形式を規定する管理ファイルへの、構造的に修正不可能な要求が止む。未登録の統治木内文書の点検は従来どおり働く。保証限界: intake の分類が正しいか（非文書と記した .md が実は出所か）は機械では判定せず、docs-curate の定例に委ねる（ADR-021 と同じ委譲先）。
