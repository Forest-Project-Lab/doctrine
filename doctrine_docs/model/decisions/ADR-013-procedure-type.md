# 手順を運ぶ型 PROC を一つだけ新設する

## 背景

頑健性評価（従来開発とビジネスの成果物584件を型登録簿へ対応付けた調査）で、時系列の段階手順（移行手順、リリース手順、切り戻し手順、障害対応手順、業務手順）を受ける型が無いと、四つの調査系統が互いに独立に判定した（20件超）。SPECの必須4節（入出力・制約・エラー時挙動・受入基準）は宣言する仕様の様式であり、段階手順が耐えない。IMPLは「仕様の正本」を入れない注意書きである。出典のS1000Dは、記述のデータモジュールと手順のデータモジュールを別型で持つ。§3.2の「新しい型を増やすのは、既存型で表せない情報が出てから」という条件を、この調査が満たした。[R8]

## 却下した選択肢

- SOP（標準作業手順書）・RUNBOOK（運用手順書）・PROCの三つの型で持つ。同義の型が並び、用語統一（R6）と最小性（R8）に反する。
- SPECの様式を緩めて手順を畳む。必須4節の機械点検が崩れ、仕様と手順の区別が消える。
- IMPLの一節で持つ。IMPLは仕様の正本を入れない注意書きであり、手順の正本になれない。

## 決定

型登録簿（§3.2）へ PROC（Procedure、手順書）を一つだけ足す。置き場所は `<domain>/procedures/`、既定statusは current、`llm_context` は task。入れるのは、目的、発動条件、前提、番号付きの手順と各段の確認点、切り戻し。入れないのは、仕様の正本と決定の理由（ADRで引く）。

## 帰結

登録簿は19型になる。テンプレートは20個になる。移行・リリース・運用・障害対応・業務の各手順が正本の置き場を得る。保証限界: リンタは型・置き場所・statusの整合を点検する（検出）。手順どおりに作業されたかは構造では点検できない（実施記録と人に委ねる）。
