# テスト不能記述の判定は監査でなく doc-review が担う

## 背景

教義 §6 の R8 行は「監査が孤児・逆孤児・テスト不能記述・`canonical_for` 衝突を一覧化する」と約束していたが、全体監査で、テスト不能記述の検査は docs-audit のどこにも実装されていないと確認した。他の三つは実装済みで、この一項だけが受入基準の宙に浮いていた。[R8][R9]

## 却下した選択肢

- 監査に機械の近似(たとえば「〜したい」「柔軟に」などの語の照合)を足す。文がテストできるかは意味の判断であり、語の照合では擬陽性と見逃しの両方が積もる。教義 §4 の役割分担(決定論は Hook、判断は Skill)にも反する。
- 約束を放置する。受入基準に実装の対応が無いままでは、体系自身が R8(テストできない記述を弾く)に違反し続ける。

## 決定

テスト不能記述の判定は doc-review(判断層)が担う。§2 の規範(各要求はテストできる形で書く。テストできない願望は要求として認めない)を doc-review の文章規範の点検項目に加える。監査(docs-audit)はテスト不能記述を検査しない。教義 §6 の R8 行から「テスト不能記述」を監査の担いから外し、doc-review の担いとして記す。

## 帰結

受入基準の全項目が実装と対応する。決定論(監査)と判断(doc-review)の分担が §4 の原則どおりに揃う。保証限界: doc-review は編集のたびの促し(review-nudge)と定例に依存するため、テスト不能記述の検出は機械的な全件保証ではない。促しの起動は hooks が支える。
