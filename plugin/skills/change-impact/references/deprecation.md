# ライフサイクルと廃止整理（§3.8）

削除は破壊ではなく一段降格とする。各段は、後から引ける記録（後継 `id` 付き）を残す。原則は可逆である。`[R4][R5][R8]`

## 段

1. 現行（`current`）。
2. 廃止（`deprecated`）。方針を撤回。事実だけを `DECIDED` の対の記録に残し、本文は LLM（大規模言語モデル）に渡さない。
3. アーカイブ（`archived`）。証跡のある本文を `<domain>/archive/` へ移し `superseded_by` を付ける。検索から除外。編集はガードが拒否。
4. git 履歴のみ。文書は消すが、コミットに撤去理由と `id` を残す。
5. 完全消去。孤児かつ証跡なしかつ再現できる一時文書だけに限る。

## 降格の不変条件

段を下げる前に、その文書への現行の依存がゼロであることを確かめる（逆参照ゼロ）。依存が残るなら、同じ変更で、依存元を後継へ張り替えてから降ろす。`dep-graph.py` を逆向きで呼ぶ。削除安全ガードが強制する（§4.2）。

## 一段だけ

一度に一段だけ降ろす。現行から直接アーカイブへ飛ばさない。各段で後継 `id` を記録し、後から引けるようにする。
