# 変更フロー全文（§3.8）

更新の順序は `ADR` → 現行 `SPEC` → 実装 → テスト → LLM（大規模言語モデル）へ渡す情報 → 廃止整理。順序を守る。「何を変えたか」（`CHANGE`／git）と「なぜ決めたか」（`ADR`）を分ける。`[R3][R4]`

## 各段が生む成果物

| ステップ | 生む成果物 | 何を書くか |
|---|---|---|
| 1 捕捉 | `CHANGE`（`proposed`） | 変更内容・理由・要求元・影響の初期見積 |
| 2 前向き | （列挙） | 波及する文書・実装・テスト |
| 3 逆向き | （列挙） | 逆参照（依存元） |
| 4 影響 | `IMPACT`（`current`） | 影響する文書・実装・テスト・工数見積。感想は書かない |
| 5 境界 | （分類） | ドメイン跨ぎは相手の ICD を通す |
| 6 決定 | `ADR`（`accepted`） | 背景・却下した選択肢・決定・帰結。根拠 |
| 7 仕様 | `SPEC`（更新） | 必須4節を保つ |
| 8 契約 | `DATA`・`API`（更新） | データ契約・接点 |
| 9 実装 | `IMPL`（更新） | 影響する部品を指す |
| 10 テスト | `TEST`（更新） | 受入基準への対応・退行観点・合否基準 |
| 11 注入 | `DECIDED`・`NONGOAL`・`WATCH`・投影 | 注入する事実を更新し、投影を描き直す |
| 12 退行 | `WATCH`（更新） | 戻してはならない事項・`review_by` |
| 13 廃止 | （降格） | 一段だけ降ろす・`superseded_by`・本文は渡さない |
| 14 仕上げ | `CHANGE`（閉じる） | 「何を」は `CHANGE`／git、「なぜ」は `ADR` |

## 順序を省かない

順序は必須である。`ADR` を先に決めないまま `SPEC` を変えない。廃止整理を先にしない。順序を守ると、根拠（`ADR`）と現行（`SPEC`）と履歴（`CHANGE`）が混ざらない。
