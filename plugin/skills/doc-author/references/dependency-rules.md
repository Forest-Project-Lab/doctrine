# 依存規則（§3.6）

## ドメイン内の依存

自由。同じドメインの現行文書どうしは `depends_on` で結んでよい。`[R7]`

## ドメイン間の依存

相手ドメインの ICD 宛だけ許す。相手の内部文書（`SPEC`・`IMPL`・`DATA` など）を直接 `depends_on` してはならない。

これにより、ICD を変えない限り、内部の変更は境界を越えない。内部が古くなっても、ドメインごと差し替えられる。

## 強制とガード

違反する `depends_on` はガードが拒否する（§4.2）。判定は構造だけで行う。相手のドメインと、相手の型が `ICD` かどうかだけを見る。相手の `status` は見ない（廃止された ICD 宛でも、書き込み時は許す。現行性は監査の関心である）。

## Write を Edit より選ぶ理由

- Write は内容が全量来るので、ガードが実行前に違反を拒否できる（`permissionDecision: "deny"`）。ディスクは変わらない。
- Edit や MultiEdit は内容の全量が来ないため、事前に拒否できない。適用後に再読して違反なら `decision: "block"` で止める。修正までディスクが一時的に不整合になる（§7）。
- だから、ドメイン間の依存を持つ文書は Write 経路へ寄せる。予防の比率を上げる。

## 解決できない依存

`depends_on` の相手が登録簿で型も引けない（接頭辞から型を判定できない）なら、ガードは安全側に倒して拒否する。相手が単に未登録（グラフに居ない）だけなら、ガードは許し、dead link は監査が拾う。
