# 復活の規則（状態の階段）

§3.8 の状態の階段に沿って、廃止・置換・アーカイブの文書がなぜ現行や LLM へ戻ってはならないかを記す。

## 状態の階段

現行 → 廃止 → アーカイブ → git だけ → 抹消。一段ずつ降ろす。逆向きに戻すには根拠を要する。

- 廃止: 方針を撤回した状態。本文は LLM へ渡さない。事実だけ DECIDED の対の記録に残す。
- アーカイブ: 証跡として倉庫へ退避した状態。アーカイブへの編集は不変性ガードが拒否する。

置換は段ではなく状態である。後継で入れ替えた状態を指し、`superseded_by` で後継を結ぶ。段を降ろすかどうかとは別に扱う。

## なぜ戻してはならないか

- 廃止・アーカイブの本文を再び現行へ戻すと、撤回した方針が復活する。これが退行である。
- 廃止本文を LLM の文脈へ載せると、撤回した方針を現行と取り違える。だから廃止本文は文脈へ渡さない（R5）。
- 状態を一段戻すには、新しい ADR で決め直す。状態だけを書き換えても、根拠の記録がなければ後から引けない。

## 正しく戻す道筋

1. 新しい ADR を起こし、背景・却下した選択肢・決定・帰結を記す。
2. DECIDED を更新し、決定事実を差し替える。
3. WATCH 項目があれば、解除の根拠を記す。
4. 文書の状態を一段だけ上げ、`updated` を記す。
